## 2. kimi-core MCP 工具清单

### 2.1 Memory 域 (tools-memory.ts)
| 工具名 | 功能 | 使用场景 |
|--------|------|----------|
| memory_write | 写入新记忆 | AI 每次对话后记录关键内容 |
| memory_reopen | 重新打开已解决记忆 | SELF_CONCERN 复发时使用 |
| memory_search | 混合搜索(语义+关键词+实体) | RAG 检索,可指定 scope/memoryType |
| memory_search_safe | 安全的非敏感检索 | 前端调用 - 硬锁定 scope=default,排除 RESTRICTED |
| memory_read | 按类型读取最近记忆 | 读取全部/指定类型记忆 |
| graph_walk | 知识图谱遍历 | 从实体/记忆出发探索关联节点 |
| memory_close | 软删除记忆 | 标记 isActive=false |

### 2.2 State/Event 域 (tools-state.ts)
| 工具名 | 功能 | 使用场景 |
|--------|------|----------|
| state_set | 设置/更新活跃状态 | 记录健康状况/心情/项目状态 |
| state_read | 读取所有活跃状态 | 上下文组装时读取 |
| state_get | 按 id 获取状态详情 | reentry 只给 summary,可拉取详情 |
| state_close | 关闭活跃状态 | 状态变更/过期 |
| event_log | 记录事件 | 通用事件日志 |
| event_read | 读取近期事件 | 查看时间线 |
| topic_create | 创建主题 | LLM 使用 |
| topic_list | 列出活跃主题 | 上下文组装时读取 |

### 2.3 Profile 域 (tools-profile.ts)
| 工具名 | 功能 | 使用场景 |
|--------|------|----------|
| profile_read | 读取核心设定 | 上下文组装 - 排除 private_ 前缀 |
| private_read | 读取私密设定 | 需要时单独调用 |
| profile_set | 设置核心设定 | 更新用户设定 |
| observation_write | 写入结构化观察 | 跨会话积累模式识别 |

| register_read | 读取所有注册设定 | 上下文组装 |

### 2.4 Entity 域 (tools-entity.ts)
| 工具名 | 功能 |
|--------|------|
| entity_write | 写入/更新实体 |
| entity_list | 列出实体 |
| entity_search | 搜索实体 |

### 2.5 Context Builder 域 (tools-reentry.ts)
| 工具名 | 功能 | 使用场景 |
|--------|------|----------|

## 3. 当前聊天上下文拼接链路

### 3.1 前端上下文组装 (ChatRoom.tsx + system-prompt.ts)

用户输入后,streamReply() 流程:

1. **SP 加载** - getSystemPrompt() -> localStorage "kimi-system-prompt", tmpl() 模板变量替换
2. **Memory 注入**(仅在 local mode) - IndexedDB 所有记忆,过滤 active + 按 order 排序,截断 5000 字
3. **kimi-core Reentry**(仅在 core mode)
   - 新对话/首次 -> fetchCoreReentryContext(threadId)
   - 间隔 >2 小时 -> fetchCoreReentryDelta(threadId)
   - 拼接: reentryCtx + \"\n\n---\n\n\" + sys.text
4. **LLM 消息组装**: system(sys.text) + history(之前**全部**对话,无裁剪) + user(当前输入)
5. **发送给 LLM** -> 流式渲染回复
6. **异步后处理**: callCoreTool(\"chat_postprocess\", {...})

### 3.2 关键文件映射
| 文件 | 功能 |
|------|------|
| src/components/chat/ChatRoom.tsx | 主聊天界面 |
| src/lib/kimi-core-client.ts | /api/core 转发客户端 |
| src/lib/tool-texts.ts | 工具调用文案池 |
| src/lib/system-prompt.ts | SP + Memory 注入逻辑 |
| src/lib/time-utils.ts | 聊天时间格式化(已修改) |
| src/lib/backend-mode.ts | 判断 local/core 模式 |
| src/app/api/core/route.ts | Edge API 路由 |

### 3.3 当前 Reentry 触发逻辑 (ChatRoom.tsx)
- 新对话/首次 -> 全量 reentry
- 间隔 >2 小时 -> 增量 reentry_delta
- 其他 -> 不调用

### 3.4 当前上下文拼接局限
- chat_read 按条数（take: 200），不按时间窗
- 无记忆检索（memory_search），全依赖 reentry 一次性注入
- 无长短对话区分
- IndexedDB 仍存 SP / session，未完全迁移到 kimi-core

## 4. 目标链路（方案讨论）

### 4.1 链路流程图

`
用户输入
  ↓
kimi-room → 发送到 kimi-core
  ↓
kimi-core 预处理（后端）：
  ├── profile_read / persona.md 读取基本设定
  ├── 当前时间注入
  ├── chat_read（按时间窗：过去 15 分钟 / 30 分钟）
  ├── 关键词检测 → 需要时调 memory_search_safe / memory_search
  ├── reentry 判定（间隔 > 2 小时或开新对话）
  │   └── 调用 reentry / reentry_delta
  └── 组装完整 context
    ↓
kimi-room 获取 context → 拼接用户输入 → 发送 LLM
  ↓
LLM 生成回复
  ↓
（输出同时）后端异步：
  ├── memory_write（记忆写入）
  ├── chat_write（对话写入）
  ├── chat_postprocess（状态/事件提取）
  ├── state_set（用户状态更新）
  └── event_log（事件记录）
`

### 4.2 关键变化对照

| 变化项 | 当前 | 目标 |
|--------|------|------|
| **SP 来源** | localStorage | kimi-core（profile_read / persona.md） |
| **对话历史** | take: 200 条 | 按时间窗（如过去 15 分钟 + 关键词命中） |
| **记忆检索** | reentry 一次性注入 | memory_search_safe（按用户输入关键词检索） |
| **长短对话** | 无区分 | 长→长模型+草稿+润色；短→短模型直接输出 |
| **reentry** | 前端判定（>2h） | 后端判定，规则相同但更可靠 |
| **IndexedDB** | 存 SP / session / dashboard 数据 | 全部走 kimi-core，移除 IndexedDB |
| **异步写入** | chat_write + chat_postprocess | + memory_write + state_set + event_log |
| **副 LLM** | 无 | 后端小模型自动检测状态变化/事件并写入 |

## 7. 待解决问题

### Q1：长短对话判断标准

短对话（早安/晚安/快速确认）和长对话（深入讨论/复杂请求/多轮推理）的判断标准是什么？

**讨论结论（暂定）**:
- 按用户输入内容判断是否需要检索（如提到"旅游"则搜旅游记录）
- 与上次 RAG 时间无关，短时间内可能换不同话题
- 尚未最终敲定判断标准的细化方案

**可选方案**:
- A：按用户输入长度（如 < 20 字 → 短对话）
- B：按对话轮数（如 < 3 轮且上次回复 > 2 小时 → 短对话）
- C：先让一个快速模型判断（耗费一次额外 LLM 调用）
- D：不做区分，统一流程

### Q2：boundary 判断是什么？

手绘图中出现的"boundary 判断"环节——是指什么边界？
- 内容合规/安全边界？
- 话题是否已完结（对话是否可以 closeout）？
- 回复是否在设定人格范围内？
- **待用户解释**
